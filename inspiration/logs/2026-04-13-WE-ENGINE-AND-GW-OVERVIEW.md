## 1. WE 是什么
 
WorkshopEngine（WE）是一个面向 AI 文字游戏/叙事游戏的**无头引擎**：它以 REST API 形式提供“游戏包导入导出、会话运行、回合循环、状态管理、工具调用、记忆系统、世界书注入、素材检索、VN 指令解析”等能力。
 
它不是一个“聊天 UI”，也不依赖浏览器执行逻辑；它把关键的“编排、解析与约束”放在服务端，以获得：
- 可复现：同一个游戏包 + 同一个会话数据，能得到稳定、可回放的运行结果。
- 可扩展：前端可以是网页、桌面、移动端；也可以存在多个不同渲染器。
- 可治理：世界书/规则/工具调用/记忆注入有清晰的审计与调试入口。

## 2. WE 解决的问题（能力图谱）
 
### 2.1 游戏包：可导入、可分享、可发布
 
WE 的核心差异之一是 **game-package.json**：把一个游戏运行所需资源打包成可迁移单元（模板、预设条目、世界书、Regex 规则、素材元数据、模型提示等），通过导入导出实现一键复现与分享。
 
### 2.2 回合循环：把“LLM 输出”变成“结构化结果”
 
每回合 WE 会产出结构化结果（示意）：
- `Narrative`：叙事文本（可用于 Markdown 渲染）
- `Options`：可选项数组（用于渲染为按钮/列表）
- `Variables`：变量快照（用于状态展示/条件判断）
- `VN`：可选的 VN 指令集解析结果（用于视觉引擎渲染）
- `TokenUsed`/错误信息：用于观测与调试
 
这意味着前端不需要“猜测 AI 在说什么”，而是消费引擎提供的稳定结构。
 
### 2.3 Prompt 编排：Pipeline + IR（可调试）
 
Prompt 组装并不是“拼字符串”，而是由多节点流水线生成一组可解释的 `PromptBlock`（带优先级），最终统一排序输出为 LLM messages。WE 同时提供 prompt-preview/snapshot 等调试能力，便于创作者定位“为什么这一轮会命中哪些世界书/记忆/预设条目”。
 
### 2.4 世界书（Worldbook）：服务端匹配与注入
 
WE 把世界书触发与递归激活放在服务端完成，并支持：
- 主/次关键词 + 逻辑门（AND/NOT）
- 正则/全词匹配 + 扫描深度
- 常驻词条、互斥分组上限、token budget 裁剪
- 变量门控（如 `var:key=value`）
 
### 2.5 变量系统：五层沙箱（可控且易 diff）
 
WE 的变量采用五级作用域沙箱（Global/Chat/Floor/Branch/Page）级联读取、合并下发。对前端与创作者来说，关键点是：
- 前端拿到的是合并后的快照（便于展示与联动）。
- 引擎可以控制“哪些阶段可写、哪些阶段只读”，避免副作用污染。
- 变量结构以“扁平 key”为主，展示分组发生在渲染层，而不是存储层。
 
### 2.6 工具系统：注册表 + 审计 + Agentic Loop
 
WE 内置可扩展的工具注册表：
- 内置工具：读写变量、检索记忆、检索素材等。
- 资源工具：让 AI 在受控范围内对 worldbook/preset/material 等资源做增量修改。
- 支持多轮工具调用（最多 N 轮），并记录工具执行审计数据。
 
### 2.7 记忆系统：可注入、可整合、可回溯
 
WE 提供记忆存取与异步整合机制：
- 记忆条目 CRUD
- 按衰减/重要度排序注入
- 异步 worker 周期性压缩/整合（把长跑会话的成本收敛）
- 记忆关系边（supports/contradicts/updates…）用于分析与一致性
 
### 2.8 VN 指令：为“视觉引擎渲染器”预留的统一协议
 
WE 支持对 VN 指令进行解析（如背景/立绘/音乐/choice 等指令），并把解析结果作为结构化字段下发。  
这与“在前端执行脚本生成 UI”是两条完全不同的路线：WE 的路线是**声明式指令 → 结构化解析 → 渲染器实现**。
 
> 这也意味着：text 游戏只是当前落地的一小部分；GW 的长期目标是以 **VNRenderer 视觉引擎**作为更高阶的呈现方式，而不是用 MVU 作为前端实现路径。

## 3. 关键架构设计（为什么这么设计）
 
下面列出对理解 WE 最关键的几项决策（完整推导请看 WorkshopEngine 仓库内的架构文档）：
 
### 3.1 PromptBlock Priority IR
 
用“带 Priority 的 PromptBlock”作为 IR，统一排序输出最终 messages。好处是可扩展、可裁剪、可调试；代价是需要清晰的约定与工具辅助（WE 提供 prompt-preview/snapshot 类端点）。
 
### 3.2 Floor / Page 双层结构（Swipe 与时间线解耦）
 
- Floor：一轮对话的时间维度（有生成/提交状态机）
- Page：同一 Floor 的多个版本（Swipe），仅一个 active
- Fork：通过复制 session 形成新时间线（易理解、易 API 化）
 
### 3.3 宏展开在后端执行
 
宏（如 `{{getvar::key}}`、`{{time}}`）在服务端 pipeline 阶段展开，保证无头引擎的一致性，并为未来副作用宏与只读阶段隔离打基础。
 
### 3.4 Worldbook 在服务端匹配
 
世界书是强耦合引擎上下文的能力（历史/变量/记忆/预算）。放到服务端可减少往返、统一裁剪与递归激活逻辑，也便于做审计与调试。

## 4. 一次回合如何工作（概念级数据流）
 
以最常见的 turn/stream 为例：
 
1) 接收输入（或选项选择）  
2) 构建 pipeline 上下文（加载模板/世界书/预设/变量/记忆/历史/工具）  
3) 组装 PromptBlock 并排序生成 messages  
4) 调用 LLM（可 SSE 流式）  
5) 解析 AI 响应（叙事/选项/VN 指令等，带回退策略）  
6) 运行后处理（Regex、宏、清理、可选 verifier）  
7) 提交回合（写 Floor/Page、写变量、写快照与审计）  
8) 下发结构化结果给前端

## 5. WE 与前端（GW）的关系：引擎与渲染器解耦
 
- WE 的边界：负责“生产结构化游戏内容与状态”，并提供可调试、可审计、可导入导出的稳定接口。  
- 前端（包括 GW）的边界：负责“把结构化结果渲染成体验”，包括 text 叙事渲染与未来 VNRenderer 视觉渲染。  
 
WE 不要求只有一个官方前端；相反，它希望存在多个渲染器与不同形态客户端，前提是：**导入接口与协议稳定**。

## 6. 开放策略与承诺（面向开发者与合作者）
 
以下是建议写进对外说明的长期原则（也建议在仓库中可追溯）：
 
1) 基础能力不商业化：核心回合循环、导入/导出、世界书、变量系统、基础游玩接口保持免费可用。  
2) 永远保留导入接口：导入是生态生命线，不以任何形式锁死。  
3) 尽可能开源最多代码：优先开源可复用的引擎层与关键协议实现；平台运营相关代码可后置评估。  
 
边界声明（减少误解）：
- 若未来出现商业化，更倾向于发生在托管/分发/增值服务，而不是封锁基础接口。  
- 对外接口与数据可导出应被视为项目原则，而不是临时决定。
